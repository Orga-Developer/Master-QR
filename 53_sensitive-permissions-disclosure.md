# Sensitive Permissions Disclosure

This document sets out the governing standard for Sensitive Permissions Disclosure within the legal architecture of Master QR. Master QR is a camera-first Android application that scans QR codes, generates QR codes, stores scan history locally in Room, supports language switching, provides share/export flows, integrates advertising and analytics components, and is distributed across mobile app stores with platform-specific compliance requirements. The document is written in advanced legal English so it can serve as a publication-ready internal policy, a store-review support artifact, and a future contract annex. It is intentionally precise about scope, permissions, data categories, and release governance, so the text can survive scrutiny across multiple jurisdictions and app ecosystems.

Applicability to Master QR: Sensitive permissions require a heightened explanation because they involve particularly private user data or device capabilities. For Master QR, the document should emphasize that camera access is used only to read QR codes, and that no microphone, contacts, or fine-grained location access is necessary for the current release. If future modules request more sensitive access, the disclosure should require a separate notice, stronger consent language, and an explicit technical justification. This policy ensures that sensitive permissions are never treated as routine or incidental. The policy must therefore distinguish between what the application actually does today and what it merely might do in a future roadmap. In the current codebase, that means recognizing camera-based scanning, QR generation, local history management, language preferences, optional exporting, advertising or analytics integrations, and the absence of broad account-based processing.

Operational requirements: the developer should document purpose limitation, legal basis where relevant, retention limits, access restrictions, vendor controls, user notice requirements, and escalation paths for exceptions. If a feature is not present, the document should not pretend that it is; instead it should state that the control is reserved for future expansion. That approach is especially valuable for store review, compliance review, and internal audit preparation.
