# Key Rotation Policy

Any cryptographic or signing material used by Master QR must be subject to rotation rules that balance continuity with security. The rotation policy should define who can trigger a key change, under what circumstances it is required, and how the application will remain operational during the transition. If keys are used for API access, analytics, or build signing, the documentation must specify whether old keys are revoked immediately, phased out gradually, or preserved temporarily for validation. Rotation events should be logged, reviewed, and linked to change-management approval. This policy is essential because long-lived keys represent a compounding risk, particularly in multi-store distribution environments where the same package may be signed or configured differently across channels.
