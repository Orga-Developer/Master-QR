# LGPD Notice

The purpose of this paper is to define a disciplined and reviewable framework for LGPD Notice in the context of Master QR. Master QR is a camera-first Android application that scans QR codes, generates QR codes, stores scan history locally in Room, supports language switching, provides share/export flows, integrates advertising and analytics components, and is distributed across mobile app stores with platform-specific compliance requirements. The document is written in advanced legal English so it can serve as a publication-ready internal policy, a store-review support artifact, and a future contract annex. It is intentionally precise about scope, permissions, data categories, and release governance, so the text can survive scrutiny across multiple jurisdictions and app ecosystems.

Applicability to Master QR: The LGPD notice should explain how Brazilian users can understand the app’s data practices, including collection categories, processing purposes, security measures, and rights to access, correction, anonymization, portability, deletion, and information about shared entities. Because the current app is functionally light and does not rely on account registration, the notice should be particularly clear about the distinction between local scan history and any third-party telemetry. It should also state that consent, legitimate interest, or another lawful basis is applied only where appropriate. The policy must therefore distinguish between what the application actually does today and what it merely might do in a future roadmap. In the current codebase, that means recognizing camera-based scanning, QR generation, local history management, language preferences, optional exporting, advertising or analytics integrations, and the absence of broad account-based processing.

Operational requirements: the developer should document purpose limitation, legal basis where relevant, retention limits, access restrictions, vendor controls, user notice requirements, and escalation paths for exceptions. If a feature is not present, the document should not pretend that it is; instead it should state that the control is reserved for future expansion. That approach is especially valuable for store review, compliance review, and internal audit preparation.

Compliance evidence, version history, and approval records should be retained so the policy can be audited against the shipped build.
