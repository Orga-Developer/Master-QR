# Build Signing and Secrets Policy

Build signing is a foundational trust control because it determines whether users and platforms can verify the origin of the distributed binary. Master QR should maintain a disciplined approach to release signing, keystore access, and secret injection through environment variables or managed vaults. No secret, API key, or signing credential should be stored in plain text inside the source tree or shipped in test artifacts without explicit purpose. The policy should also describe how fallback debug signing is constrained, how secret rotation is handled, and how broken or missing secrets affect the release pipeline. This protects distribution integrity, reduces the risk of unauthorized repackaging, and supports forensic accountability if a build is later disputed.
