# Government Request Policy

This instrument formalizes the rules for Government Request Policy and aligns them with the actual behavior of the application. Master QR is a camera-first Android application that scans QR codes, generates QR codes, stores scan history locally in Room, supports language switching, provides share/export flows, integrates advertising and analytics components, and is distributed across mobile app stores with platform-specific compliance requirements. The document is written in advanced legal English so it can serve as a publication-ready internal policy, a store-review support artifact, and a future contract annex. It is intentionally precise about scope, permissions, data categories, and release governance, so the text can survive scrutiny across multiple jurisdictions and app ecosystems.

Applicability to Master QR: This policy explains how the developer or platform operator responds to lawful requests from government authorities. For Master QR, the document should clarify the distinction between data stored locally on the device, data held by third-party services, and data under the developer’s control. It should also set expectations for legal review, narrow disclosure, minimum necessary production, and user notification where legally permitted. This is important for a product that intentionally minimizes account-based data collection, because request handling must mirror the limited data architecture. The policy must therefore distinguish between what the application actually does today and what it merely might do in a future roadmap. In the current codebase, that means recognizing camera-based scanning, QR generation, local history management, language preferences, optional exporting, advertising or analytics integrations, and the absence of broad account-based processing.

Operational requirements: the developer should document purpose limitation, legal basis where relevant, retention limits, access restrictions, vendor controls, user notice requirements, and escalation paths for exceptions. If a feature is not present, the document should not pretend that it is; instead it should state that the control is reserved for future expansion. That approach is especially valuable for store review, compliance review, and internal audit preparation.

The policy should be revisited whenever the app adds new screens, new permissions, or new processing purposes.
